# CLAUDE.md

This file defines how AI agents should work in this repository.

## Project Context

- Stack: Laravel 12 (PHP 8.3), Inertia v2, Vue 3, Tailwind 3.
- This repository is application code (`grow-place`), not a tooling-only repository.
- Toolkit bundles are intentionally included in `toolkits/` and should be used.

## Mandatory Rule Sources

Agents must use the following toolkit paths:

- `toolkits/rubx-agentic-tools/conduct/backend/**` — Laravel coding standards (PHP, architecture, security, logging, etc.)
- `toolkits/rubx-agentic-tools/claude-code/backend/**` — rubx-laravel skills

**Do NOT use** `toolkits/rubx-agentic-tools/conduct/frontend/**` — it is Nuxt 3–specific and does not apply to this project. Vue + Inertia rules are in `.cursor/rules/vue-inertia.mdc`.

And also follow repository-native rules/config:

- `.cursor/rules/**`
- Existing project conventions in current codebase

## Rule Priority (Conflict Resolution)

When rules conflict, apply this order:

1. Direct user instruction in current task
2. Repository-native rules (`.cursor/rules/**`)
3. This file (`CLAUDE.md`)
4. Toolkit files in `toolkits/**`

If conflict still remains, prefer framework-native implementation for this repo:

- Laravel + Vue conventions win over Go-specific recommendations.
- Use toolkit process/checklist guidance, but implement code in the local stack style.

## Toolkit

The shared toolkit lives in `toolkits/rubx-agentic-tools/` (git submodule).

## Standard Working Flow

1. Read task and identify affected Laravel/Vue areas.
2. Draft/update plan in markdown:
   - Product plan: `docs/plans/product/*.md`
   - Dev plan: `docs/plans/dev/*.md`
3. Implement smallest correct change aligned with local conventions.
4. Self-review using toolkit checklists and project rules.
5. Run required local validation commands for touched areas.
6. Report what changed, why, and remaining risks.

## Database Safety — STRICTLY FORBIDDEN

The following commands **MUST NEVER be run** by any agent or developer on a local or shared environment without explicit written approval from the user in the current session:

- `php artisan migrate:fresh` — drops ALL tables and reruns all migrations (destroys all data)
- `php artisan migrate:refresh` — same destruction, with optional seeding
- `php artisan migrate:reset` — rolls back all migrations (destroys all data)
- `php artisan db:wipe` — drops all tables, views, and types

**Always use `php artisan migrate` only** — runs pending migrations without touching existing data.

If a success criterion in a plan says "run migrate:fresh", interpret it as "run on a dedicated empty test DB" — never on the active local database.

A PreToolUse hook in `.claude/settings.json` will block these commands automatically.

## Database Snapshot For Agents

- Prefer `database/schema.snapshot.json` as the first source for current database structure.
- This file is auto-generated by `php artisan db:schema-snapshot`.
- After running migrations, the snapshot is refreshed automatically via the migration-ended event hook.
